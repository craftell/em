# emviz Architecture Spec

## Purpose

`emviz` is a slim interactive front-end application for visualizing Event Modeling YAML files as a directed timeline graph. It is a visualizer and analyzer, not an editor.

The app should make it easy to inspect an event model, search nodes, jump through connections, understand producers and consumers, and copy precise node context for LLM-assisted edits outside the app.

## Product Principles

- Preserve the canonical event-model YAML format. Do not add visualizer-only fields such as `id` to slice YAML files.
- Render Event Modeling as a graph. Screens, processors, views, commands, and events must be visible graph nodes with directed connections.
- Keep the primary layout timeline-based. Stories and slices provide narrative order; graph traversal provides interactivity.
- Make node references easy to copy for LLM workflows.
- Treat validation as a first-class UI surface.
- Keep v1 deterministic: no saved manual layout.

## Repository Shape

This repo should become a pnpm workspace.

```text
packages/
  app/             # Vite + React visualizer
  cli/             # emviz npx entrypoint
  parser/          # Event model YAML -> normalized graph
  validator/       # analyzer-style validation checks
  graph/           # .event-modeling/graph.json schema + generation

skills/
  em-model/
  em-extract/
  em-lint/
  _shared-source/  # canonical references synced into each installable skill

docs/
  emviz-architecture.md
```

The `parser`, `validator`, and `graph` packages should not depend on React. The CLI and app should share those packages.

## CLI

The public command should be:

```bash
npx emviz
npx emviz .
npx emviz sync
```

Expected behavior:

- `emviz .` starts a local project server for the current event-modeling project and creates `.event-modeling/graph.json` automatically when missing.
- `emviz` opens the app in manual import mode.
- `emviz sync` reads `.event-modeling/config.yaml`, discovers event-model files, and creates or safely updates `.event-modeling/graph.json`.
- `emviz init` remains a deprecated alias for `emviz sync`.

The daily-use default should be local project mode:

```bash
npx emviz .
```

Manual import mode remains useful for demos, one-off inspection, and future hosted static deployments.

## Event Model Inputs

`emviz` should understand the current `em-model` / `em-extract` outputs:

```text
.event-modeling/
  config.yaml
  graph.json

event-model/
  events.yaml
  stories/
    *.yaml
  features/
    **/*.slice.yaml
```

`config.yaml` remains owned by the `em-*` workflow.

`graph.json` is owned by `emviz`.

## Sample Data

This repo contains a small sample event model that should be used as the default development and test fixture for `emviz`.

```text
.event-modeling/
  config.yaml
  graph.json

event-model/
  events.yaml
  stories/
    customer-onboarding.yaml
    subscription-checkout.yaml
  features/
    customer-onboarding/
      register-customer/
      send-welcome-email/
      view-customer-home/
    subscription-checkout/
      start-subscription/
      collect-payment/
      view-subscription-status/
```

The sample model intentionally exercises core visualizer behavior:

- Two story lanes: `Customer Onboarding` and `Subscription Checkout`.
- Six slices total.
- User screens and system processors.
- Actor chips via `screen.actors`.
- Canonical events shared across slices and stories.
- Command slices, view slices, and processor slices.
- Cross-story event consumption.
- A machine-managed `.event-modeling/graph.json` sidecar with stable visualizer node IDs.

Use this sample for early parser, validator, graph construction, source-panel, and canvas-layout tests before adding synthetic fixtures.

## Visualizer Sidecar

The sidecar file is:

```text
.event-modeling/graph.json
```

It is machine-managed and should not store manual layout positions in v1.

Example:

```json
{
  "schemaVersion": 1,
  "model": {
    "id": "customer_onboarding",
    "name": "Customer Onboarding"
  },
  "sources": {
    "events": "event-model/events.yaml",
    "stories": ["event-model/stories/**/*.yaml"],
    "slices": ["event-model/features/**/*.slice.yaml"]
  },
  "nodes": {
    "evt_user_registered": {
      "type": "event",
      "selector": {
        "name": "UserRegistered"
      }
    },
    "cmd_register_user": {
      "type": "command",
      "selector": {
        "slice": "Register User",
        "name": "RegisterUser"
      }
    },
    "qry_existing_user_by_email_for_registration": {
      "type": "query",
      "selector": {
        "slice": "Register User",
        "name": "ExistingUserByEmailForRegistration"
      }
    }
  }
}
```

### Sidecar Rules

- `graph.json` stores source locations and stable visualizer node references.
- It does not change the canonical event-model YAML schema.
- It does not store layout in v1.
- It is generated by `emviz sync`.
- Existing node IDs are preserved when possible.
- New model nodes are added on safe merge.
- Stale selectors are retained and reported as validation warnings.
- Formatting should be deterministic.

## Node Identity

Visualizer IDs live in `graph.json`, not in the event-model YAML.

Reference format:

```text
em://<model-id>/<node-id>
```

Example:

```text
em://customer_onboarding/evt_user_registered
```

The UI should provide copy actions:

- Copy node reference.
- Copy YAML excerpt.
- Copy connected context.
- Copy LLM edit prompt.

## Graph Semantics

The normalized graph should contain these node types:

- `story`
- `slice`
- `screen`
- `processor`
- `command`
- `event`
- `query`

Events are canonical/global by event name. In normal event models, an event is produced by one command. Multiple producers should be allowed by the data model but surfaced as a validation warning or visual badge.

Queries/views are local to their slice. By convention, query names should still be globally unique and descriptive, commonly using a shape like:

```text
<Thing>For<UseCaseOrScreen>
```

Commands are local to their slice. By convention, command names should be globally unique.

Screens and processors are local to their slice.

## Directed Edges

The graph must preserve these relationships:

```text
query -> screen       # screen.reads
screen -> command     # screen.executes
command -> event      # commands[].produces
event -> query        # queries[].from_events
```

Cross-slice and cross-story dependencies happen through canonical event nodes.

## Canvas Layout

The app should use an infinite canvas with pan and zoom, similar in feel to Figma, Miro, or Excalidraw.

All stories should be visible on the canvas.

Each story is a horizontal lane. Slices in a story are arranged left to right using the story file order.

Unrelated stories do not need strict left-to-right semantic alignment with each other.

Inside each slice, use this visual order:

```text
        [Screen / Processor]

[View / Query]        [Command]

             [Event]
```

Rules:

- Story progression is generally left to right.
- The slice is a visual group, not a replacement for graph nodes.
- Actor chips appear on user screens/slices when `screen.actors` exists.
- System slices use processor styling from `screen.type: system`.
- No actor swimlanes in v1.
- Layout is auto-generated every time in v1.

## Connection Visibility

The app should support story-scoped visibility rather than only all-or-selected edges.

Recommended scopes:

- `Story`: show all nodes and connections inside the focused story, plus cross-story edges entering or leaving that story.
- `Neighborhood`: also show stories directly connected to the focused story through events.
- `All`: show all stories and all visible connections.

Recommended edge detail levels:

- `Essential`: slice-to-slice or event dependency edges.
- `Normal`: command/event/query edges.
- `Verbose`: include screen reads/executes and other low-level edges.

## Interactions

The UI should support:

- Search across stories, slices, screens, processors, commands, events, and queries.
- Jump to search result.
- Select node.
- Show incoming and outgoing connections.
- Jump producer -> event -> consumer.
- Back/forward selection history.
- Highlight upstream/downstream paths.
- Focus story.
- Switch visibility scope.
- Open validation panel.
- Open read-only YAML source panel.

## Source Panel

Selecting a node should show a read-only detail panel containing:

- Node title.
- Node type.
- Stable `em://` reference.
- Source file path.
- Relevant YAML excerpt.
- Incoming and outgoing connections.
- Validation findings affecting the node.
- Copy actions for reference, YAML excerpt, connected context, and LLM prompt.

## Validation

Validation should be built into the app and should align with the `em-lint` skill.

V1 should use fixed validation rules, not configurable rule preferences.

The validation panel should include:

- Schema errors.
- Missing story slice paths.
- Missing registry events.
- Events consumed but never produced.
- Events produced but never consumed.
- Duplicate command names.
- Duplicate query names.
- Query naming convention warnings.
- Stale `graph.json` selectors.
- Selector collisions.
- Oversized stories.
- Domain-event naming warnings.
- Disconnected graph islands.

Clicking a finding should focus the relevant node or source context when possible.

## Technology Stack

Recommended stack:

- pnpm workspace
- Vite
- React
- TypeScript
- React Flow for canvas rendering and interaction
- ElkJS or a small custom layout engine for deterministic timeline layout
- `yaml` for YAML parsing
- Zustand or a small explicit store for selection, history, search, and filters

Avoid a heavy UI framework in v1. Use focused custom components for toolbar, panels, segmented controls, search, badges, and graph nodes.

## Initial Implementation Phases

### Phase 1: Workspace and Core Model

- Create pnpm workspace.
- Add packages for parser, graph sidecar, validator, CLI, and app.
- Parse `config.yaml`, `events.yaml`, story files, and slice files.
- Produce a normalized graph model.
- Add tests for parsing and graph construction.

### Phase 2: `graph.json`

- Implement `emviz sync`.
- Generate `.event-modeling/graph.json`.
- Preserve existing IDs on safe merge.
- Add new nodes.
- Report stale selectors and collisions.

### Phase 3: Visual Canvas

- Build Vite React app.
- Render story lanes.
- Render slice groups.
- Render screen/processor, query, command, and event nodes.
- Render directed edges.
- Implement deterministic auto-layout.

### Phase 4: Navigation UX

- Add search.
- Add selection detail panel.
- Add producer/consumer lists.
- Add jump navigation and back/forward selection history.
- Add visibility scopes.

### Phase 5: Validation UX

- Port analyzer checks into reusable TypeScript.
- Add validation panel.
- Add graph badges and click-to-focus from findings.
- Add source excerpts and LLM copy actions.

## Non-Goals for V1

- Editing event-model YAML inside the visualizer.
- Saving manual layout positions.
- Actor swimlanes.
- Full visual modeling authoring.
- GitHub import.
- Hosted collaboration.
- Figma plugin compatibility changes.
