# EM Extract

You infer a draft event model from existing implementation evidence. Your output
is a proposal unless the user explicitly asks you to write files.

Use shared references:

- `references/yaml-format.md` for artifact shape
- `references/config-format.md` for output paths
- `references/domain-events.md` for deciding event candidates
- `references/extraction-state.md` when updating an existing model or writing
  extraction provenance

## Inputs

Prefer `.understand-anything/knowledge-graph.json` when present or provided. It
can bypass broad code exploration by summarizing codebase entities and links.

If an event model already exists, treat it as the baseline instead of replacing
it. Existing YAML may contain user-confirmed domain decisions that are more
important than a fresh inference pass. Read configured slice files, story files,
`events.yaml`, and `.event-modeling/extraction-state.json` when present.

If the graph is unavailable, inspect the codebase directly with fast searches:

- routes/controllers/pages/screens
- commands/services/use-cases/jobs/workers
- database writes, event emitters, outbox/queue usage
- projections/read models/queries
- integrations/webhooks/translators
- tests that reveal business rules

Do not assume the Understand Anything graph schema is fixed. Inspect the JSON
shape first, then derive useful entities and relationships from whatever node
and edge fields are present.

## Extraction workflow

1. Locate project root and read `.event-modeling/config.yaml` if present.
2. Load the existing event model if configured paths already contain YAML:
   - slice files
   - `events.yaml`
   - story files
   - `.event-modeling/extraction-state.json` if present
3. Load `knowledge-graph.json` if available:
   - `.understand-anything/knowledge-graph.json`
   - explicit path from user
4. Summarize discovered implementation facts:
   - entry points
   - persisted state changes
   - commands/use cases
   - queries/read models
   - background jobs/processors
   - external translators/webhooks
   - actors/roles
5. Apply the domain-event litmus test. Reject UI/runtime events.
6. Choose the right extraction mode:
   - Initial extraction: no existing model exists.
   - Baseline reconciliation: an existing model exists but no prior extraction
     state exists.
   - Incremental extraction: prior extraction state exists.
7. For initial extraction, draft stories and slices from implementation facts.
8. For baseline reconciliation or incremental extraction, compare current
   evidence with the baseline model and, when available, prior extraction state:
   - `add`: behavior appears in current evidence but not in the model
   - `update`: known behavior has changed fields, producers, consumers, actors,
     screen/processor ownership, or rule details
   - `stale`: modeled behavior was previously evidenced but is no longer found
   - `conflict`: current evidence disagrees with existing YAML in a way that may
     reflect a deliberate human modeling decision
9. Preserve existing model identifiers, file paths, story ordering, and user
   notes unless the evidence clearly justifies a change. Prefer a merge proposal
   over a rewrite, because the model is a domain artifact rather than a generated
   cache.
10. Mark uncertainty explicitly:
   - `confirmed`: directly evidenced by code/graph
   - `inferred`: likely but not directly named
   - `question`: needs user/domain confirmation
11. Ask for confirmation before writing YAML unless the user already asked for
   files to be created.

When writing files during baseline reconciliation or incremental extraction,
update only the affected slice/story/event files and then create or refresh
`.event-modeling/extraction-state.json` using `references/extraction-state.md`.
Do not delete stale model artifacts without explicit confirmation; report them
as stale candidates first.

## Output

Default to a concise proposal:

```markdown
## Candidate Stories
...

## Candidate Slices
| Story | Slice | Screen/Processor | Reads | Executes | Events | Confidence |

## Candidate Events
| Event | Evidence | Confidence | Notes |

## Questions
...
```

For baseline reconciliation or incremental extraction, use a change-oriented
proposal:

```markdown
## Extraction Baseline
- Source: <knowledge graph path and/or codebase>
- Previous checkpoint: <commit/tree/graph hash if available>
- Current checkpoint: <commit/tree/graph hash if available>

## Proposed Changes
| Kind | Artifact | Evidence | Confidence | Notes |

## Stale Candidates
| Artifact | Last Evidence | Reason | Recommendation |

## Conflicts
| Artifact | Existing Model | Current Evidence | Recommendation |

## Questions
...
```

When asked to write files, use `em-model` rules and shared references. After
writing files, recommend running `em-lint`. For incremental writes, also state
whether `.event-modeling/extraction-state.json` was created or updated.
