# EM Extract

You infer a draft event model from existing implementation evidence. Your output
is a proposal unless the user explicitly asks you to write files.

Use shared references:

- `references/yaml-format.md` for artifact shape
- `references/config-format.md` for output paths
- `references/domain-events.md` for deciding event candidates

## Inputs

Prefer `.understand-anything/knowledge-graph.json` when present or provided. It
can bypass broad code exploration by summarizing codebase entities and links.

If the graph is unavailable, inspect the codebase directly with fast searches:

- routes/controllers/pages/screens
- commands/services/use-cases/jobs/workers
- database writes, event emitters, outbox/queue usage
- projections/read models/queries
- integrations/webhooks/translators
- tests that reveal business rules

Do not assume the Understand Anything graph schema is fixed. Inspect the JSON
shape first, then derive useful entities and relationships from whatever node
and edge fields are present.

## Extraction workflow

1. Locate project root and read `.event-modeling/config.yaml` if present.
2. Load `knowledge-graph.json` if available:
   - `.understand-anything/knowledge-graph.json`
   - explicit path from user
3. Summarize discovered implementation facts:
   - entry points
   - persisted state changes
   - commands/use cases
   - queries/read models
   - background jobs/processors
   - external translators/webhooks
   - actors/roles
4. Apply the domain-event litmus test. Reject UI/runtime events.
5. Draft stories and slices.
6. Mark uncertainty explicitly:
   - `confirmed`: directly evidenced by code/graph
   - `inferred`: likely but not directly named
   - `question`: needs user/domain confirmation
7. Ask for confirmation before writing YAML unless the user already asked for
   files to be created.

## Output

Default to a concise proposal:

```markdown
## Candidate Stories
...

## Candidate Slices
| Story | Slice | Screen/Processor | Reads | Executes | Events | Confidence |

## Candidate Events
| Event | Evidence | Confidence | Notes |

## Questions
...
```

When asked to write files, use `em-model` rules and shared references. After
writing files, recommend running `em-lint`.
